# 页面信息架构

## 1. 目标

群策的前端不是传统“聊天窗口”，而是一个以群聊为主形态的多智能体工作台。

前端需要同时满足三件事：

- 先完成节点配对和身份配置
- 再进入像微信群一样的群聊主界面
- 在群聊过程中能看到智能体状态、执行状态和错误反馈

首版信息架构要优先服务 MVP，不做过度拆分。

## 2. 设计原则

- 中文优先，所有主要导航和状态文案先用中文
- 主场景优先，用户进入系统后的主要停留页应该是群聊页，而不是配置后台
- 配置和聊天分离，避免聊天页承担过多管理表单
- 实时状态可见，节点在线、发言中断、执行失败都要能在界面上第一时间看到
- 可控自动化，群聊支持智能体自动接话，但界面上必须能随时暂停或终止

## 3. 全局导航结构

推荐一级结构如下：

```text
/setup
/chats
/chats/:chatId
/settings/nodes
/settings/personas
/settings/runtime
```

首版不建议把“房间设置”做成单独一级导航，直接放在群聊页内的右侧抽屉或房间详情弹层里。

## 4. 首次使用流程

首次进入系统时，优先走 `setup` 引导，而不是直接落到空白聊天页。

引导顺序：

1. 创建或登录管理员账号
2. 生成本地节点配对码
3. 检查 `Local Agent` 是否上线
4. 创建第一个智能体身份
5. 创建第一个群聊房间
6. 进入群聊页

如果用户已经完成配对并至少有一个聊天，则首页直接跳转到最近使用的 `/chats` 或 `/chats/:chatId`。

## 5. 页面清单

### 5.1 `/setup`

用途：初始化系统，建立“本地节点可用 -> 身份可用 -> 房间可用”的第一条闭环。

页面模块：

- 顶部产品区：产品名、简短说明、当前步骤
- 步骤区：配对节点、创建身份、创建房间
- 结果区：当前节点在线状态、已创建身份数量、是否可进入群聊

主要动作：

- 生成配对码
- 刷新节点状态
- 新建身份
- 新建房间
- 完成并进入群聊

空状态：

- 没有本地节点时显示安装和启动 `Local Agent` 的引导
- 节点未上线时显示“等待连接”而不是报错

### 5.2 `/chats`

用途：房间列表页，也是无当前房间时的过渡页。

页面模块：

- 左侧房间列表
- 中间房间摘要区
- 右上角新建房间按钮

每个房间卡片展示：

- 房间名称
- 最近一条消息摘要
- 最近活跃时间
- 当前在线成员数
- 是否存在运行中的发言轮次

主要动作：

- 打开房间
- 新建房间
- 归档房间

### 5.3 `/chats/:chatId`

这是产品的核心页面，布局建议固定为三栏。

### 左栏

- 房间列表
- 搜索框
- 新建房间按钮
- 设置入口

### 中栏

- 房间头部
  - 房间名称
  - 当前讨论模式
  - 在线成员数量
  - 节点在线状态
- 消息流
  - 用户消息
  - 智能体消息
  - 系统消息
  - 运行状态消息
- 输入区
  - 文本输入框
  - `@` 提及智能体
  - 发送
  - 开始自由讨论
  - 暂停讨论
  - 终止当前轮次

### 右栏

- 房间成员列表
- 每个智能体当前状态
  - 待命
  - 运行中
  - 离线
  - 出错
- 房间规则摘要
  - 点名模式 / 受控自由讨论
  - 最大轮数
  - 每轮最多发言人数
  - 冷却时间
- 最近执行日志入口

### 消息类型

消息流至少要支持以下视觉类型：

- `用户消息`
- `智能体消息`
- `系统通知`
- `流式消息`
- `失败消息`
- `中断消息`

其中“流式消息”要在同一条消息气泡里持续增量更新，不要每个 chunk 单独成条。

### 主要交互

用户在群聊页可以做的关键动作：

- 输入普通消息并发送
- `@` 某个智能体定向提问
- 启动一轮受控自由讨论
- 暂停当前讨论
- 终止正在执行的发言
- 打开某条消息关联的运行详情
- 编辑房间规则

### 5.4 `/settings/nodes`

用途：管理本地节点和连接状态。

页面模块：

- 节点列表
- 当前在线状态
- 最近心跳时间
- 系统信息
  - 主机名
  - 平台
  - 架构
  - Agent 版本
- 运行能力
  - 最大并发数
  - 当前 worker 数

主要动作：

- 生成新的配对码
- 重命名节点
- 禁用节点
- 查看节点最近错误

### 5.5 `/settings/personas`

用途：管理不同的智能体身份。

页面模块：

- 身份列表
- 身份编辑表单
- 预览卡片

每个身份至少配置：

- 名称
- 头像
- 角色说明
- 系统提示词
- 发言风格
- 是否默认自动发言
- 单次发言超时

主要动作：

- 创建身份
- 复制身份
- 启用 / 停用
- 调整默认发言参数

### 5.6 `/settings/runtime`

用途：系统运行参数设置。

页面模块：

- 默认超时
- 默认上下文窗口大小
- 默认自由讨论轮数
- 默认每轮最大发言人数
- 日志保留天数

首版不要把运行设置做得过细，避免把产品变成运维后台。

## 6. 关键页面流程

### 6.1 发送一条点名消息

1. 用户在输入框输入内容
2. 用户通过 `@` 选择一个智能体
3. 前端调用 HTTP 接口创建消息
4. Server 落库后向当前房间广播新消息
5. Server 调度对应 `turn`
6. 前端通过 WebSocket 收到流式输出并更新同一条消息
7. 消息完成后状态变为“已完成”

### 6.2 启动自由讨论

1. 用户点击“开始自由讨论”
2. 前端弹出轻量确认框
3. 用户设置最大轮数和每轮发言人数
4. Server 创建讨论轮次
5. 前端显示系统消息“自由讨论已开始”
6. 后续发言按照编排规则逐轮追加

### 6.3 中断当前发言

1. 用户点击某条流式消息上的“终止”
2. 前端调用 HTTP 中断接口
3. Server 向对应 `Local Agent` 下发中断指令
4. 前端收到状态更新
5. 消息显示为“已中断”

## 7. 前端状态模型

前端至少维护以下几类实时状态：

- 当前登录用户
- 节点在线状态
- 房间列表和最近活跃状态
- 当前房间消息流
- 当前运行中的 `turn`
- 智能体在线和忙碌状态

建议：

- 配置类数据走 `TanStack Query`
- 群聊消息和实时状态走 WebSocket 增量同步
- 不要把所有实时状态都塞进全局 store

## 8. 空状态和错误状态

首版必须明确设计以下状态：

- 没有配对节点
- 节点离线
- 没有身份
- 没有房间
- 消息发送失败
- 发言超时
- 执行中断
- 本地 `codex` 启动失败

这些状态不能只靠 toast 提示，需要在页面内有可见反馈。

## 9. 推荐前端目录结构

```text
console/src
  app/
  pages/
    setup/
    chats/
    settings/
  components/
    chat-list/
    chat-timeline/
    chat-composer/
    member-panel/
    run-drawer/
    persona-form/
    node-card/
  features/
    chats/
    messages/
    personas/
    nodes/
    runtime/
  lib/
    api/
    ws/
    utils/
```

这个结构的核心思想是：

- 页面负责组织布局
- `features` 管业务数据
- `components` 管可复用 UI
- `lib/ws` 单独封装实时连接
